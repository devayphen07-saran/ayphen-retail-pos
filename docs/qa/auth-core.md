QA/BA Test Cases: Auth Core (password/crypto, rate limiting, login-attempt cleanup)

Scope: `apps/backend/src/auth/core/`
- `crypto.service.ts` — JWT sign/verify, snapshot HMAC signing, Ed25519 device-signature verification, AES-256-GCM cache encryption, SHA-256 token hashing, canonical JSON.

## 1. Feature understanding (BA)

### 1.1 Actors
- **Unauthenticated caller** (any IP) — hitting login/OTP/signup endpoints.
- **Phone-holder** — a real or attacker-controlled phone number receiving/entering OTPs.
- **Authenticated mobile principal** — holds an access JWT + device session, subject to `verifyJwt`/blacklist/replay checks (guard itself is out of scope; `verifyJwt`/`decodeOwnJwtClaims` are in scope).
- **System/cron** — the login-attempts-cleanup scheduler, running unattended, unattended by a human.
- **Operator/SRE** — depends on Redis availability; observes fallback behavior and cleanup stats.

### 1.2 What each unit does

**CryptoService**
- `signJwt(sub, deviceSessionId, permissionsVersion)` — HS256 JWT, claims `{sub, jti (random UUID), type:'access', deviceSessionId, pv, iat, exp}`. `exp = iat + ACCESS_TOKEN_TTL_SECONDS` (default **900s / 15 min**).
- `decodeOwnJwtClaims(token)` — **unverified** base64url decode of a token this process just signed itself (never for externally-supplied tokens — that's a documented contract, not enforced by types).
- `verifyJwt(token)` — verifies signature+expiry via `jose.jwtVerify` (algorithm pinned to HS256), then re-validates claim shape with a Zod schema (`sub`, `jti`, `type`, `deviceSessionId`, `pv`, `iat`, `exp` — `type` kept as loose `string()` so the guard, not this method, raises `INVALID_TOKEN_TYPE`). Any `jose` failure (bad sig, expired, malformed, wrong alg) is swallowed and rethrown uniformly as `UnauthorizedError(TOKEN_INVALID, 401, "Invalid or expired access token")` — never leaks *why* it failed. A shape mismatch throws the same class with `"Malformed access token"`.
- `signSnapshot` / `verifySnapshot` — HMAC-SHA256 over canonical JSON (documented as an **Ed25519 placeholder** until a real key pair is provisioned). `verifySnapshot` recomputes the expected HMAC and compares with `timingSafeEqual`, additionally short-circuiting on length mismatch before calling `timingSafeEqual` (buffers of different length would otherwise throw).
- `verifyDeviceSignature(publicKeyBase64, challenge, signatureHex)` — real Ed25519 via WebCrypto `SubtleCrypto`. **Any** error (bad base64/hex, wrong key length, unsupported curve, verify() returning false) is caught and normalized to `false` — never throws.
- `encryptJson` / `decryptJson` — AES-256-GCM, random 12-byte IV per call, key = `sha256("cache-encryption:" + JWT_REFRESH_SECRET)` derived once at `onModuleInit`, never persisted. Ciphertext blob = `iv(12) || authTag(16) || ct`. `decryptJson` throws on any tamper (ciphertext, tag, or IV) or on wrong key (e.g. after `JWT_REFRESH_SECRET` rotation) — callers are documented to treat that as a cache miss, not a hard error.
- `hashToken` — plain SHA-256 hex digest (used for refresh-token-at-rest hashing and OTP-at-rest hashing — **not** a slow/salted hash). Appropriate for refresh tokens because they are high-entropy random values. For OTPs (6-digit, ~900k possible values — **not** high entropy) the same unsalted SHA-256 is still safe, but for different reasons: a 5-attempt verify lockout (`OTP_MAX_ATTEMPTS`, `otp-request.repository.ts`) makes online brute-force infeasible, the hash is stored only in Redis with a 300s TTL (`OTP_TTL_SECONDS`) and is never persisted to Postgres, so there is no durable hash column to exfiltrate for offline cracking.
- `canonicalJson` — recursively sorts object keys (arrays keep order) so semantically-identical objects always serialize identically; feeds `signSnapshot`.

**PasswordService**
- `hash(password)` — argon2id, `memoryCost=65536` (64 MiB), `timeCost=3`, `parallelism=4`. Enforces `assertLength` first (max length from config, default **1024** chars) — throws `UnprocessableError(PASSWORD_TOO_LONG, 422)` if exceeded.
- `verify(hash, password)` — same length guard, then `argon2.verify`; **any** exception (malformed hash, wrong argon2 variant, decode error) is caught and mapped to `false` rather than propagating — a corrupt stored hash fails safe (rejects) rather than 500ing.
- `needsRehash(hash)` — true if the stored hash's parameters differ from the current `ARGON2_OPTIONS` (i.e. the cost parameters were bumped since the hash was created).
- **Note:** as of this codebase, no caller in `auth/mobile/**` invokes `PasswordService` — login is phone+OTP only (see §7 Open Questions). It is still fully in scope here as a unit, since it's part of Auth Core and will presumably back a future email/password or web path.

**RateLimitService + RateLimitRepository**
- Two fixed-window limiters, both **enforced primarily via Redis**, degrading to an atomic Postgres counter only when Redis is unreachable (never fails open):
  - `checkIpLimit(ip)` — key `rl:ip:{ip}`, window **60s**, limit `IP_MAX_ATTEMPTS` (default **100**).
  - `checkPhoneOtpLimit(phone)` — key `rl:otp:{phone}`, window **300s (5 min)**, limit `OTP_MAX_ATTEMPTS` (default **5**).
- Redis path: single Lua script does `INCR` then `EXPIRE key windowSeconds` **only if this INCR created the key** (`c == 1`) — a fixed window anchored to the *first request in the window*, sliding forward each time the key fully expires and a new one is created.
- Fallback path (Redis unreachable — any exception from `redis.eval`, including timeout): Postgres `INSERT ... ON CONFLICT (key, windowStart) DO UPDATE SET count = count + 1`, atomic per row. **Its window is computed differently**: `windowStart = floor(now / windowMs) * windowMs` — an **epoch-aligned** bucket, not "first request in window." This is a real semantic difference between the two paths (see §7 Open Questions / §4 edge cases).
- Both paths block when the **post-increment** count exceeds the limit (`count > limit`), i.e. the request that pushes the count from `limit` to `limit+1` is the first one rejected — the `limit`-th request itself is allowed.
- On block: throws `AppException(RATE_LIMIT_EXCEEDED, <message>, 429)` — distinct message per limiter (IP vs phone-OTP wording).
- `recordAttempt(entry)` — pure audit-trail insert into `login_attempts` (`ip`, optional `userId`/`email`/`phone`, `purpose`, `success`). **Does not feed back into either limiter** — the counters and the audit table are entirely independent data paths. A failure to write the audit row does not block or affect enforcement (and isn't retried).
- Real callers observed: `otp-request.service.ts` (IP + phone check before minting an OTP, records `success:false` after send), `auth-login.service.ts` stage two (IP check, then phone check *after* OTP-request lookup and account-status checks, before verifying the OTP code; records failure on bad OTP, success after full transaction commits), `auth-signup.service.ts` (same IP+phone pattern), `step-up.service.ts` (phone check only, no IP check, before an OTP-based step-up).

**LoginAttemptsCleanupService**
- Registers one cron job at `onModuleInit` (`CronJob` from the `cron` package) on schedule `CRON_LOGIN_ATTEMPTS_CLEANUP` (default `'30 3 * * *'` — 03:30 server time, daily), started immediately, named `'login-attempts-cleanup'` in Nest's `SchedulerRegistry`.
- `cleanOldAttempts()`: re-entrancy-guarded by an in-memory `isRunning` boolean (skips silently — no log — if already running); computes `cutoff = now - LOGIN_ATTEMPTS_RETENTION_DAYS * 24h` (default **30 days**); deletes `login_attempts` rows with `createdAt < cutoff` (strict less-than — a row exactly at the cutoff instant survives that run) and `rate_limit_fallback_counters` rows with `windowStart < cutoff` (same cutoff, no separate config for the fallback table); updates `stats` (`lastRunAt`, `lastDurationMs`, `lastRemovedCount`, `error`) **only on the success path**; on error, only `stats.error` is set (`lastRunAt` is left at its previous value, i.e. a failed run does not update "last run" timestamp) and `isRunning` is always reset in `finally`.
- Single unbounded `DELETE ... WHERE created_at < cutoff` with `.returning()` — no batching/chunking/limit.

### 1.3 Business rules / invariants extracted
1. Access tokens are valid for exactly `ACCESS_TOKEN_TTL_SECONDS` (900s) from issuance; any signature/shape/expiry failure is a uniform 401 `TOKEN_INVALID`, never a distinguishing error.
2. `decodeOwnJwtClaims` must only ever be called on a token this process itself just signed — no verification is performed.
3. Snapshot signatures use constant-time comparison; must reject both a tampered payload and a tampered signature.
4. Device-signature verification never throws — always resolves to a boolean.
5. Cache-encrypted payloads must fail closed (throw) on any tamper or wrong key — never silently return corrupted plaintext.
6. Passwords over `MAX_PASSWORD_LENGTH` (1024) are rejected before hashing/verifying (DoS guard against giant inputs to argon2).
7. A wrong password (or a corrupt/foreign-format hash) must return `false`, never throw, from `verify`.
8. IP requests are capped at 100/60s; phone-OTP requests capped at 5/300s — both enforced whether Redis is up or down.
9. Rate-limit enforcement must never fail open — a Redis error degrades to a DB-backed counter, it never skips the check.
10. The audit trail (`login_attempts`) is independent of enforcement; it must not be relied upon to block anything, and its own failure must not block a login/OTP flow (not explicitly caught by a try/catch at the call sites observed — see §7 open question on whether an audit-write failure should be swallowed).
11. Retention cleanup only deletes rows strictly older than the configured window and never runs two overlapping cleanups concurrently.

### 1.4 Assumptions flagged
- A1: "at the limit is allowed, limit+1 is rejected" is inferred from `count > limit` — treated as the intended (not accidental) boundary.
- A2: The Redis-window-vs-DB-window semantic mismatch (first-request-anchored vs epoch-aligned) is treated as a real, currently-unflagged behavioral inconsistency rather than an intentional design choice — flagged as an open question, tested as current behavior.
- A3: `PasswordService` is tested as a standalone unit per the request scope even though no current mobile call site invokes it.
- A4: "IP"/"phone" strings passed into the limiters are assumed pre-validated/normalized by callers (e.g. E.164 phone format) — the service itself does no format validation, so malformed/unnormalized keys are tested as a negative/robustness case, not a rule violation.

## 3. Test cases

### 3.1 CryptoService — JWT (CRY-JWT-xx)

**CRY-JWT-01 — Sign then verify round-trip**
Area: happy | Criticality: Critical | Traces to: Rule 1
Preconditions: `JWT_ACCESS_SECRET` configured (≥32 chars), service initialized (`onModuleInit` ran).
Input: `sub='user-123'`, `deviceSessionId='sess-abc'`, `permissionsVersion=4`.
Steps: 1) `signJwt(sub, deviceSessionId, 4)`. 2) `verifyJwt(token)`.
Expected: Resolves with `{sub:'user-123', jti:<uuid>, type:'access', deviceSessionId:'sess-abc', pv:4, iat, exp}`; `exp - iat === 900`.

**CRY-JWT-02 — Expired token rejected**
Area: rule (violated) | Criticality: Critical | Traces to: Rule 1
Preconditions: Sign a token, then advance/mock system clock (or use a short-lived test config) past `exp`.
Steps: `verifyJwt(expiredToken)`.
Expected: Throws `UnauthorizedError` with `errorCode=TOKEN_INVALID`, `statusCode=401`, message `"Invalid or expired access token"` — no distinguishing "expired" vs other failure reason leaked.

**CRY-JWT-03 — Tampered payload rejected**
Area: negative | Criticality: Critical | Traces to: Rule 1
Steps: Sign a token, flip one base64 character in the payload segment, call `verifyJwt`.
Expected: Same uniform `TOKEN_INVALID` 401 (signature check fails before shape check ever runs).

**CRY-JWT-04 — Wrong secret rejected**
Area: negative | Criticality: Critical | Traces to: Rule 1
Preconditions: Token signed under secret A; service instance configured with secret B (simulates a secret-rotation mismatch across processes/pods).
Expected: `TOKEN_INVALID` 401 — never a 500.

**CRY-JWT-05 — Malformed token string**
Area: negative | Criticality: High | Traces to: Rule 1
Input: `"not-a-jwt"`, `""`, `"a.b"` (missing segment), `"a.b.c.d"` (extra segment).
Expected: Every variant throws `TOKEN_INVALID` 401, never an unhandled exception/500.

**CRY-JWT-06 — Algorithm confusion attempt**
Area: negative (security) | Criticality: Critical | Traces to: Rule 1
Steps: Craft a token with `alg: 'none'` or `alg: 'HS512'` and a payload that would otherwise pass shape validation.
Expected: `jwtVerify` rejects because `algorithms:['HS256']` is pinned — `TOKEN_INVALID` 401. Verify the code path never reaches the Zod schema for a wrong-alg token.

**CRY-JWT-07 — Valid signature, wrong claim shape**
Area: rule (violated) | Criticality: High | Traces to: Rule 1
Steps: Sign (with the same secret, out of band) a JWT missing `deviceSessionId` or with `pv` as a string instead of number.
Expected: `jwtVerify` succeeds but Zod `safeParse` fails → `UnauthorizedError(TOKEN_INVALID, "Malformed access token")`, 401.

**CRY-JWT-08 — `type` claim is loose at this layer**
Area: rule (satisfied) | Criticality: Medium | Traces to: `verifyJwt`'s documented contract (guard owns type enforcement)
Steps: Sign a token whose `type` were something other than `'access'` (not producible via `signJwt` itself — construct manually with same secret) and call `verifyJwt` directly (not through the guard).
Expected: `verifyJwt` succeeds and returns the claims as-is (`type` is `z.string()`, not a literal) — confirms type-enforcement is the caller's (guard's) job, not this method's. Note in report: this is by design per the source comment; a unit test asserting the opposite would be a false regression signal.

**CRY-JWT-09 — `jti` uniqueness across repeated signs**
Area: rule | Criticality: Medium | Traces to: replay-protection dependency
Steps: Call `signJwt` twice with identical arguments back-to-back.
Expected: Two different `jti` values (random UUID per call) — downstream blacklist/replay logic depends on this.

**CRY-JWT-10 — `decodeOwnJwtClaims` on a just-signed token**
Area: happy | Criticality: High | Traces to: Rule 2
Steps: `token = signJwt(...)`; `decodeOwnJwtClaims(token)`.
Expected: Returns `{jti, exp}` matching what `verifyJwt` would also report — no signature check performed (function completes even if the secret were subsequently rotated, since it never validates).

**CRY-JWT-11 — `decodeOwnJwtClaims` on external/garbage input**
Area: negative | Criticality: Medium | Traces to: Rule 2 (documented misuse boundary)
Input: A token from a different signer, or `"a.b.c"` with `b` not valid base64/JSON.
Expected: Throws a raw `SyntaxError`/`TypeError` (JSON.parse or Buffer decode failure) — **not** wrapped in `AppException`. Flag: any accidental future call site passing untrusted input here will surface an unhandled 500, not a clean 401. (See §7.)

**CRY-JWT-12 — Access-token TTL boundary (899s vs 900s vs 901s)**
Area: boundary | Criticality: High | Traces to: Rule 1
Steps: Sign, then verify at `iat+899s` (valid), `iat+900s` (== exp, JWT `exp` semantics are exclusive per RFC — verify actual `jose` behavior), `iat+901s` (expired).
Expected: 899s passes; 901s fails as `TOKEN_INVALID`; document the exact boundary behavior observed at `t=900s` (jose treats `exp` as the instant after which the token is invalid, i.e. `now >= exp` fails) as the authoritative answer for this exact-boundary case.

### 3.2 CryptoService — snapshot signing (CRY-SNAP-xx)

**CRY-SNAP-01 — Sign/verify happy path**
Area: happy | Criticality: High | Traces to: Rule 3
Steps: `canonical = canonicalJson({b:2,a:1})`; `sig = signSnapshot(canonical)`; `verifySnapshot(canonical, sig)`.
Expected: `true`.

**CRY-SNAP-02 — Verify fails on tampered payload**
Area: rule (violated) | Criticality: Critical | Traces to: Rule 3
Steps: Sign canonical A, then call `verifySnapshot(canonicalB, sigOfA)` where B differs by one field.
Expected: `false`.

**CRY-SNAP-03 — Verify fails on tampered signature**
Area: negative | Criticality: Critical | Traces to: Rule 3
Steps: Flip one hex character of a valid signature, call `verifySnapshot`.
Expected: `false` (not a throw).

**CRY-SNAP-04 — Signature of different length never throws**
Area: boundary | Criticality: High | Traces to: Rule 3 (guards `timingSafeEqual`'s equal-length requirement)
Input: `signature = "ab"` (2 hex chars) vs a real 64-char SHA-256 hex signature.
Expected: `verifySnapshot` returns `false` cleanly (length check short-circuits before `timingSafeEqual`), never throws `RangeError`.

**CRY-SNAP-05 — Empty/garbage signature string**
Area: negative | Criticality: Medium
Input: `signature = ""`, `signature = "not-hex!!"`.
Expected: `Buffer.from(sig,'hex')` on non-hex input silently truncates/produces a short buffer in Node — verify actual behavior; either way must resolve `false`, never throw.

**CRY-SNAP-06 — `canonicalJson` key-order independence**
Area: rule | Criticality: High | Traces to: Rule 3 (signature stability)
Input: `{a:1,b:{y:2,x:1}}` vs `{b:{x:1,y:2},a:1}`.
Expected: `canonicalJson` produces byte-identical strings for both → identical `signSnapshot` output — a snapshot rebuilt from the same logical data always re-verifies even if key insertion order differs across code paths.

**CRY-SNAP-07 — `canonicalJson` array order is preserved (not sorted)**
Area: boundary | Criticality: Medium
Input: `{list:[3,1,2]}` vs `{list:[1,2,3]}`.
Expected: These serialize to **different** canonical strings (arrays are not sorted) — confirms array order is semantically significant and must be preserved by producers.

**CRY-SNAP-08 — `canonicalJson` with `null`, `undefined`, nested arrays-of-objects**
Area: edge | Criticality: Medium
Input: `{a: null, b: undefined, c: [{z:1,y:2}]}`.
Expected: Matches `JSON.stringify` semantics for `undefined` (key dropped) and `null` (kept as `null`); nested object keys inside array elements are still sorted.

Input: `password = 'Str0ng!Passw0rd#2026'`.
Steps: `hash = await hash(password)`; `await verify(hash, password)`.

**PWD-02 — Wrong password rejected**

**PWD-03 — Same password hashed twice yields different hashes**
Area: rule | Criticality: High | Traces to: argon2 salting
Steps: `hash(password)` twice.
Expected: Two different hash strings (random salt per hash); both independently `verify(true)` against the original password.

**PWD-04 — Corrupt/foreign-format hash fails safe**
Area: negative | Criticality: Critical | Traces to: Rule 7
Input: `hash = 'not-an-argon2-hash'`, or a bcrypt hash (`$2b$...`).
Steps: `verify(hash, 'anything')`.
Expected: Resolves `false` — the internal `try/catch` swallows argon2's parse error; never throws/500s.

**PWD-05 — Password at exact max length (1024 chars) accepted**
Area: boundary | Criticality: High | Traces to: Rule 6, `MAX_PASSWORD_LENGTH` boundary
Input: A 1024-character password.
Expected: `hash()` succeeds; `verify()` against the same 1024-char string succeeds.

**PWD-06 — Password one over max length (1025 chars) rejected**
Area: boundary/negative | Criticality: High | Traces to: Rule 6
Input: A 1025-character password.
Steps: `hash(password)`.
Expected: Throws `UnprocessableError` with `errorCode=PASSWORD_TOO_LONG`, `statusCode=422`, before argon2 is ever invoked (DoS guard).

**PWD-07 — `verify` also enforces the same length guard**
Area: rule | Criticality: Medium | Traces to: Rule 6
Steps: `verify(someHash, veryLong1025CharString)`.
Expected: Throws `PASSWORD_TOO_LONG` 422 — same as `hash`, so an attacker can't bypass the guard by only ever calling `verify` with oversized input.

**PWD-08 — Empty-string password**
Area: boundary | Criticality: Medium | Traces to: Rule 6/7

**RL-FB-06 — DB fallback insert failure (Postgres also down)**

## 4. Edge-case scenarios (§5 checklist — sneaky/real-world)

**EDGE-01 — Empty/zero: brand-new store, zero login attempts ever recorded**
Traces to: CLN-02/CLN-05. First-ever cleanup run on a fresh deployment with an empty `login_attempts` table.
Expected: No error; `lastRemovedCount = 0`.

**EDGE-02 — First-run/migration: table just created, cron fires before any traffic**
Traces to: CLN-01/CLN-05. Cron fires at 03:30 before the app has served a single login.
Expected: Runs cleanly against empty tables.

**EDGE-03 — Maximum/overflow: OTP/IP counters at exactly the configured max, then one more of each simultaneously**
Traces to: RL-IP-02/03, RL-PHONE-02/03 combined — an IP that is *also* the phone-limit boundary in the same instant (e.g. the 100th IP call for an IP is simultaneously the 6th phone-OTP call for the phone tied to that request).
Expected: Each limiter evaluates and rejects/accepts independently based purely on its own counter — a phone-limit rejection must not consume/affect the IP counter and vice versa (both `enforce()` calls run to completion for their own key regardless of the other's outcome, since they're two separate `await` calls in the caller, e.g. `auth-login.service.ts` calling `checkIpLimit` then `checkPhoneOtpLimit` sequentially — if the IP check throws first, the phone counter for this attempt is never incremented at all, meaning a request that fails the IP check leaves the phone-OTP counter untouched. Verify this exact sequencing is intended.).

**EDGE-04 — Decimals/rounding: N/A** — no monetary or fractional-quantity logic in this module (crypto/rate-limit/cleanup are integer-counter and boolean domains). Explicitly out of scope; not a gap.

**EDGE-05 — Duplicate/repeat: identical login attempt submitted twice within milliseconds (double-tap)**
Traces to: RL-FB-03 concurrency case, applied to the Redis-happy-path too.
Steps: Fire two concurrent `checkPhoneOtpLimit(samePhone)` calls (simulating a mobile client's accidental double-submit) when currently at 4/5.
Expected: Redis `INCR` is atomic — one call sees count 5 (passes), the other sees count 5 or 6 depending on ordering, but never both see the same pre-increment value; if both land within the 5 allowed, both pass; if the double-tap is the 5th+6th, exactly one is rejected — no double-admission of over-limit traffic.

**EDGE-06 — Out-of-order: `verifyJwt` called on a token signed by a request that started earlier but arrives later than a subsequent token for the same session**
Traces to: CRY-JWT-01/09. Two access tokens for the same `deviceSessionId` in flight (e.g. client retried a stale request after already refreshing).
Expected: `verifyJwt` alone has no concept of ordering/staleness — both verify successfully if both are unexpired (jti-based staleness/replacement is the guard's/session-repo's job, not `CryptoService`'s). Confirms this unit's responsibility boundary; document that ordering correctness is enforced one layer up (`MobileJwtGuard`'s `currentJti` check), out of this module's scope but worth cross-referencing so nobody assumes `CryptoService` alone prevents replay of a superseded token.

**EDGE-07 — Concurrent identical: two devices for the same user hit `checkIpLimit` for the same NATed office IP simultaneously (many cashiers, one store IP)**
Traces to: RL-IP-05 inverted — same-IP shared-tenancy case explicitly called out in the source comment ("real abuse limit is per-phone; 100/min here is deliberately loose for carrier-grade NAT/shared store IPs").
Steps: 20 different store devices behind the same public IP each make a legitimate login attempt within 60s.
Expected: All 20 pass comfortably under the 100 cap — confirms the generous IP limit is sized for exactly this legitimate multi-device-per-IP retail scenario, not just as an oversight. Add a regression case at 100 legitimate concurrent devices (unlikely in a single store, but plausible across a shared corporate NAT/franchise HQ) to confirm the cap doesn't wrongly block a big legitimate burst.

**EDGE-08 — Offline→sync: N/A for rate-limit/crypto (server-side, stateless per-request)** — this module has no offline/client-queue concept; explicitly out of scope. Cross-reference: mobile client-side OTP resend cooldown UX is out of this module (lives in `otp-request.service.ts` / client), not tested here.

**EDGE-09 — Permission/subscription change mid-flow: `JWT_ACCESS_SECRET` rotated while tokens are in flight**
Traces to: CRY-JWT-04. Ops rotates the access-token secret (e.g. as an incident response to a suspected leak) while users hold valid, unexpired tokens signed under the old secret.
Expected: Every in-flight token immediately fails `verifyJwt` with `TOKEN_INVALID` 401 the next request — i.e. secret rotation is equivalent to an instant global logout of all sessions (no grace/dual-secret verification path exists in `onModuleInit`, which reads only `this.config.jwtAccessSecret` once). Flag as an operational fact to document in the incident-response runbook, not a defect — but confirm this is the intended blast radius before an incident actually happens (§7).

**EDGE-10 — Abandonment/interruption: cleanup cron killed mid-DELETE (process crash/OOM/deploy restart)**
Traces to: CLN-07/CLN-09. Simulate the process exiting mid-statement (not just a thrown JS error, but a hard process kill).
Expected: Postgres itself rolls back the incomplete `DELETE` (it's a single statement/implicit transaction) — no partial deletion. On restart, `onModuleInit` re-registers the cron fresh (`isRunning` reinitializes to `false` in memory) and the next scheduled tick runs normally; no stuck "isRunning forever" state survives a process restart (it's in-memory only, not persisted) — confirm this is understood as a feature (self-healing on restart) not a hidden assumption.

**EDGE-11 — Time: clock skew between the JWT-signing process and the verifying process (multi-instance deployment)**
Traces to: CRY-JWT-02/12. Instance A signs a token with a system clock 5s ahead of instance B's clock; B verifies shortly after issuance.
Expected: `jose` typically allows a small default clock-tolerance window; confirm the actual tolerance (none is explicitly configured here — verify whether `jwtVerify`'s default `clockTolerance` of 0 applies) — a meaningfully-skewed fleet clock could cause spurious `TOKEN_INVALID` right after issuance for tokens whose `iat` appears to be "in the future" relative to the verifying node. Recommend NTP-sync verification in ops; flag as a cross-cutting infra dependency, not a code defect, but worth an explicit `clockTolerance` config decision (§7).

**EDGE-12 — Time: DST transition during the nightly cleanup window**
Traces to: CLN-01/CLN-12. Cron fires at `03:30` local time on a DST-transition night (a night with 23 or 25 hours).
Expected: Confirm whether the underlying `cron` library's local-time evaluation skips or double-fires 03:30 on the transition night (varies by library/timezone). Document actual behavior; if the deployment region observes DST (verify — many retail deployments in India, this repo's apparent primary market given `+91` phone examples, do **not** observe DST, which would make this a non-issue in practice — confirm target deployment regions in §7).

**EDGE-13 — Connectivity transitions: Redis flaps rapidly (up/down/up) within a single rate-limit window**
Traces to: RL-FB-04/05, expanded to 3+ transitions within one 60s window.
Expected: Each transition independently either uses Redis's (persistent, TTL-backed) counter or creates/increments a DB row for the current epoch-aligned bucket; total real limit enforcement during a flapping period is strictly looser (never stricter) than the nominal 100/60s, consistent with RL-FB-04's documented finding — never causes an incorrectly *stricter* block of legitimate traffic.

**EDGE-14 — Long/unusual input: extremely long phone/IP string passed as the rate-limit key**
Traces to: RL-PHONE-07/RL-IP-07. `phone` = a 500-character garbage string (e.g. a client bug concatenating fields).
Expected: Works mechanically (Redis/Postgres both accept arbitrarily long text keys) but produces a rate-limit bucket nobody will ever legitimately hit again — no functional break, but note that unbounded key length is caller-input, not validated here; recommend length-capping at the call site/DTO layer.

**EDGE-15 — Unicode/RTL in password (extends PWD-09) at exactly the length boundary**
Traces to: PWD-05/06/09. A 1024-**character** password containing multi-byte emoji (each 1 JS "char" via surrogate pairs may count as 2 in `.length`, or 4-byte UTF-8 on the wire) — verify `assertLength`'s `.length` check is JS UTF-16 code-unit length, and that this doesn't produce a mismatch against what argon2 hashes (UTF-8 bytes), i.e. a 1024 UTF-16-length string full of emoji could be far more than 1024 *bytes*, and conversely a 1024-byte-limit intent is not actually what's enforced. Flag as a units clarification worth confirming with product/security (§7) — the guard is a DoS mitigation, and byte-length vs. code-unit-length matters for how effective that mitigation actually is against a maximal multi-byte payload.

**EDGE-16 — State edge: `verify()` called against a hash for a user who was deleted/soft-deleted between hash-fetch and verify-call**
Traces to: PWD-01/04. Not directly this module's concern (deletion/soft-delete is `UserRepository`'s job) — but confirm `PasswordService.verify` itself is a pure function of `(hash, password)` with no DB/state lookup, so it behaves identically regardless of the account's current status; the *caller* is responsible for checking account state before/after. Cross-reference `auth-login.service.ts`'s `isBlocked`/`status` checks as the actual enforcement point (already covered in the mobile-auth QA scope, not duplicated here).

**EDGE-17 — Device/platform: `verifyDeviceSignature` receiving an Android-vs-iOS differently-encoded public key (e.g. one platform base64-encodes with URL-safe alphabet, the other standard)**
Traces to: CRY-DEV-04/06. Feed a URL-safe-base64 key (using `-`/`_` instead of `+`/`/`) into a decoder path that expects standard base64.
Expected: `Buffer.from(str,'base64')` in Node is lenient and generally decodes URL-safe base64 correctly too (verify empirically) — if it does *not* silently succeed with wrong bytes, confirm the failure mode is a clean `false`, not a subtle wrong-but-"valid"-looking key that verifies against the wrong data. Cross-check with the actual mobile client's encoding convention (§7).

## 7. Open questions (need product/dev confirmation)

1. **Redis-window vs. DB-fallback-window semantics differ** (RL-FB-02/04/05): the Redis path anchors each window to the first request that creates the key; the Postgres fallback anchors to an absolute epoch-aligned bucket (`floor(now/windowMs)*windowMs`). Is this intentional? If not, should the fallback instead read/write a "window start" the same way the Redis TTL implies one (e.g. by also storing a window-start timestamp keyed off first-hit rather than epoch alignment)?
2. **Total outage (Redis down *and* Postgres fallback insert failing)** (RL-FB-06): currently propagates as an unhandled exception (likely 500) rather than either failing open or returning a clean 429/503. Is a 500 the desired behavior during a full infra outage, or should there be an explicit fail-closed 503 with a clear error code?
3. **`decodeOwnJwtClaims` misuse safety** (CRY-JWT-11): should this throw a structured `AppException` instead of a raw `SyntaxError`/`TypeError` if ever accidentally called on external input, as defense-in-depth against a future refactor that violates the "only call on self-signed tokens" contract?
4. **`PasswordService` is currently unused** by any live call site in `auth/mobile/**` — is a password/email-based login or web-admin login path planned soon (so this suite stays relevant and gets wired into a real DTO/controller with its own rate-limiting/lockout), or is it legacy/scaffold that should be removed or clearly marked `@deprecated`/`@future` to avoid confusing future maintainers?
5. **`login_attempts.email` / `recordAttempt({email})`** is schema- and interface-supported but has zero real callers, and there is no `checkEmailLimit` counterpart to `checkIpLimit`/`checkPhoneOtpLimit`. Is an email-based signup/login/limiter path planned? If yes, note it needs its own rate-limit method with its own window/threshold decision.
6. **Password length guard units** (EDGE-15): is `MAX_PASSWORD_LENGTH=1024` intended as a character count (current `.length` behavior, UTF-16 code units) or a byte-count DoS bound? For a maximal-emoji password these diverge meaningfully. Worth an explicit decision if the DoS-mitigation intent is byte-based.
7. **JWT clock tolerance** (EDGE-11): no explicit `clockTolerance` is passed to `jwtVerify`. Given a horizontally-scaled backend, is fleet clock skew (NTP-based, should be sub-second) considered negligible, or should a small explicit tolerance (e.g. 5s) be configured defensively?
8. **`LOGIN_ATTEMPTS_RETENTION_DAYS` has no lower-bound validation** (CLN-10) — should the env schema enforce `.min(1)` (or a sane floor like 7) to prevent an operator typo (e.g. `0` or a negative value) from mass-deleting the audit trail on the very next cron tick?
9. **Cleanup stats on a failed run** (CLN-08) — `lastRunAt`/`lastDurationMs`/`lastRemovedCount` are not updated when a run errors, so a dashboard reading only those three fields (without checking `error`) can show a stale "healthy" timestamp indefinitely after repeated failures. Should a failed run also stamp `lastRunAt`/`lastDurationMs` (with `lastRemovedCount` left at its last-known-good value, or set to `null`/`-1` to make failure visually obvious)?
10. **Deployment timezone/DST** (EDGE-12) — confirm the production host/container timezone for the cron scheduler (the `+91` phone examples throughout the mobile auth code suggest an India-primary market, which does not observe DST — if true, EDGE-12 is a non-issue in practice, but should be confirmed rather than assumed for any region the backend might expand to).
11. **Phone/IP normalization is entirely the caller's responsibility** (RL-PHONE-07, RL-IP-07/14) — should `RateLimitService` itself normalize (e.g. E.164-normalize phone, or IP-canonicalize) before keying, as defense-in-depth against a future caller forgetting to normalize and accidentally creating an easy rate-limit bypass?
